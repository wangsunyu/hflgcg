## 目标
阶段 4 的核心目标不是单独把 CAS 页面展示出来，而是完成完整登录闭环：
1. 获取 `appId`
2. 构建 CAS 登录地址
3. 加载 CAS H5 登录页
4. 截获登录成功后的 `mobile_code`
5. 用 `mobile_code` 换取 token
6. 用 token 拉取用户信息
7. 保存登录态与 Cookie
8. 登录成功后切换到主路由承接界面

## CAS 登录流程

### 1. 获取 appId
启动登录页后，先请求：
- `APIConfig.getAppId`

当接口返回成功且 `data` 非空时，得到当前登录所需的 `appId`。

### 2. 构建 CAS 地址
将 `appId` 拼到 CAS 认证地址：
- `https://ids-hfit-edu-cn-s.hfit.edu.cn/authserver/mobile/auth?appId={appId}`

当前项目通过 `URLBuilder.buildCASAuthURL(appId:)` 统一构建。

### 3. 加载 CAS 页面
`LoginView` 使用 `WebView` 加载上一步生成的 URL。
登录页展示逻辑包括：
- 准备中加载态
- WebView 页面
- 页面加载失败后的重试态
- 登录中遮罩

## mobile_code 截获方式
旧项目在 `webView:didCommitNavigation:` 中监听 URL 变化；当前 SwiftUI 重构版通过 `WKNavigationDelegate` 的 URL 变更回调上抛给 `LoginViewModel.handleURLChange(_:)`。

截获策略：
- 只处理主框架 URL
- 检查 URL 查询参数中是否存在 `mobile_code`
- 若 queryItems 未直接解析出值，则退回到完整字符串中查找 `mobile_code=`
- 如 URL 中还包含 `#`，仅截取 `#` 前的 code
- 使用 `hasStartedLoginExchange` 与 `capturedMobileCode` 防止重复换 token

## token 与用户信息调用顺序
登录闭环的接口顺序固定如下：

### 1. getTokenByCode
请求：
- `APIConfig.getTokenByCode`

参数：
- `mobile_code=<截获到的 code>`

成功后返回业务 token。

### 2. userProfile
请求：
- `APIConfig.userProfile`

参数：
- `token=<上一步拿到的 token>`

成功后返回用户资料，包括：
- `userCode`
- `userName`
- `userType`
- `refreshToken`
- `fileAddress`

### 3. 保存登录信息
将 token 与用户信息整理为 `LoginInfo`，再调用：
- `UserManager.shared.saveLoginInfo(_:)`

保存完成后，`UserManager.shared.isLoggedIn` 会被持久化为 `true`，供下次启动分流使用。

## Cookie 保存与恢复要求

### 保存时机
CAS 登录成功、拿到用户信息后，需要把当前 `WKWebView` 内的 Cookie 持久化。
当前项目通过：
- `CookieManager.shared.saveCookies(from:completion:)`

### 恢复时机
任何承载校园 H5 的 `WebView`，都应在**首次执行 `load(URLRequest)` 前**先恢复 Cookie。
当前项目统一在 `Features/WebContainer/WebView.swift` 中处理：
1. 首次 `updateUIView`
2. 调用 `CookieManager.shared.restoreCookies(to:completion:)`
3. Cookie 恢复完成后再加载目标 URL

这样登录页与后续 H5 容器页可以共用同一套 Cookie 链路，避免每个页面单独恢复。

## 启动分流要求
阶段 4 闭环不仅包含登录成功，还包含下次启动时的分流：
- `UserManager.shared.isLoggedIn == false` → Splash 结束后进入 `.login`
- `UserManager.shared.isLoggedIn == true` → Splash 结束后进入 `.main`

该分流由：
- `HFITCampus/HFITCampus/Features/Splash/SplashView.swift`

统一负责。

## 阶段 4 验收口径
满足以下链路时，可认为阶段 4 完整闭环成立：
1. 冷启动先进入 Splash
2. 未登录用户：Splash → Login
3. Login 正常获取 `appId` 并加载 CAS 页面
4. 成功截获 `mobile_code`
5. `getTokenByCode` 成功换取 token
6. `userProfile` 成功返回用户信息
7. `UserManager` 保存登录信息成功
8. Cookie 已保存
9. 登录成功后进入 `.main`
10. 重启 App 后可根据 `isLoggedIn` 跳过登录页
11. WebView 再次加载时 Cookie 已恢复，会话不中断
